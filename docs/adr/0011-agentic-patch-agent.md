# Agentic Patch Agent (Phase 3)

Turn `patch_agent` from a single-shot generator into a bounded read+edit ReAct
agent that works on a per-run **staging copy** of the BSP repo: it reads the exact
regions it needs, applies edits, re-reads the updated content, and iterates. The
node's output is a real `git diff` of the staging copy. This removes the whole class
of "the writer can't see the file" failures.

Status: **Proposed** (developed on branch `agentic-patch-agent`, merged after it
passes).

## Context

Today `patch_agent` is a single `chat_completion` with no tools. It only sees the
summarized `repo_inspection`, then emits search/replace blocks (ADR-0010) that are
applied later. This has a structural flaw — **the actor that writes the edit cannot
read the file** — which produced a chain of bugs and does not scale:

```text
- evidence summarized away file content  -> model hallucinated SEARCH text
- one bad block fails the whole patch    -> all-or-nothing on multi-location edits
- Source-excerpts 40k cap                -> big / many-location edits lose coverage
- blocks applied against a summary        -> ordering / overlap errors on one file
```

Robust agents (Claude Code, Cursor) avoid this by having the **same** agent read and
edit, one edit at a time, each verified against the live file. That is phase 3.

## Decision

`patch_agent` becomes a bounded ReAct agent (`langchain.agents.create_agent`, as in
ADR-0006) that operates on a **staging git worktree** created from the run's base:

```text
tools (read-only): grep_repo, read_file        (existing, sandboxed)
tool  (write):     edit_file(path, search, replace, replace_all=false)
                   -> str_replace on the STAGING copy only (reuses edit_tools logic)

loop: read_file / grep_repo -> edit_file -> read_file (verify) -> ... (bounded)
output: git diff of the staging worktree = a real, base-generated diff
```

Key properties:
- **The writer can read.** Each edit is chosen after reading the actual current
  text, so SEARCH text is copied, never hallucinated.
- **Content scales with edits, not files.** The agent reads the window it edits, not
  a giant prompt; no 40k evidence cap in the write path.
- **Per-edit validation, no all-or-nothing.** `edit_file` matches against the *live
  staging content*; a bad edit returns an actionable error to the agent, which fixes
  just that edit in-loop. Ordering/overlap resolve naturally because the next read
  sees prior edits.
- **The produced diff is reliable.** It is generated by real edits on the base, so
  it applies cleanly — no fuzzy hunk normalization needed.

### Staging + safety

- Per run, create a staging worktree of the BSP repo at the current base
  (`git worktree add` at the base commit, or a temp checkout). `edit_file` is
  sandboxed to the staging path; the real working tree is **never** touched during
  patch generation.
- The node computes `git -C <staging> diff` -> `diff_text`.
- After the graph run, the staging worktree is removed.

### Pipeline integration

```text
inspect_repo (ReAct evidence)   unchanged (localization hints for the patch agent)
patch_agent (ReAct on staging)  reads+edits staging -> diff_text
validate_patch                  diff is real -> git apply --check sanity only
code_review_agent / human_review unchanged (see the diff)
apply_patch                     git apply diff_text to the real tree (reliable,
                                same base) after approval
publish                         unchanged
```

`edit_file` reuses the `edit_tools` str_replace core (unique match / REPLACE_ALL /
actionable errors), now called per-edit against live staging content instead of a
batch validated against a summary.

## Boundaries

- Real BSP working tree is untouched until `apply_patch` after human approval; all
  patch-generation edits happen on the staging copy.
- The agent gets read + edit(staging) only — no commit / push / shell.
- Bounded by `recursion_limit` and per-run staging cleanup; a non-converging loop
  ends with whatever diff exists (or none -> NO_PATCH), routed as today.
- Requires reliable model tool-calling (verified for the configured Qwen model in
  the ADR-0006 probe); the write path now depends on it too.

## Consequences

Multi-location and large-file edits become reliable: no hallucinated SEARCH, no
all-or-nothing, no evidence-coverage cap in the write path, and the emitted diff is
guaranteed to apply. The cost is staging-worktree lifecycle management, a second
LangChain agent (the patch agent, like inspect_repo), and a heavier reliance on the
model's tool-calling. Because this reworks the core patch path, it is developed and
tested on a branch and merged to main only after it passes.

Supersedes the single-shot half of ADR-0010: the search/replace *primitive*
(`edit_tools`) is kept and reused by `edit_file`; the single-shot block-emission +
batch validation is replaced by the agentic read+edit loop.
